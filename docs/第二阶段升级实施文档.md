# 电商评论情感分析系统第二阶段升级实施文档

## 1. 第二阶段目标

第二阶段不再只证明系统“能判断正负面”，而是把系统升级为“能解释用户为什么满意或不满意”的评论洞察平台。

升级目标分为三个层级：

1. **基础增强**：保留第一阶段单条分析、批量分析、历史查询、统计导出能力，补齐性能、缓存、错误处理和部署稳定性。
2. **展示增强**：新增正负向高频词云、价格/物流/质量评价雷达图，让用户可以直观看到商品卖点和槽点。
3. **智能增强**：新增一键总结接口，自动生成商品优点、缺点和购买建议。该能力作为选做增强，必须提供规则兜底，不能强依赖 LLM。

## 2. 第二阶段功能优先级

| 优先级 | 功能 | 是否必做 | 负责人 | 说明 |
| --- | --- | --- | --- | --- |
| P0 | Docker、接口、前端、数据库稳定运行 | 必做 | B、D | 保证第一阶段功能不回归 |
| P0 | 批量任务状态、错误行、缓存命中率 | 必做 | B、D | 让系统可观测、可测试 |
| P1 | 正/负向高频词云 | 必做 | A、B、C | 展示用户喜欢什么、吐槽什么 |
| P1 | 价格/物流/质量评价雷达图 | 必做 | A、B、C | 展示商品多维度口碑 |
| P1 | 模型评估与性能报告 | 必做 | C、D | 输出 Accuracy、F1、响应时间、缓存效果 |

第二阶段最低交付标准：**P0 + P1 全部完成**。

## 3. 新增核心功能设计

### 3.1 正/负向高频词云

目标：把评论中的核心词汇提取出来，分别展示正向评论和负向评论的高频词。

用户价值：

- 正向词云：展示用户认可点，例如“物流快”“质量好”“便宜”“包装好”。
- 负向词云：展示用户槽点，例如“破损”“贵”“发货慢”“异味”“掉色”。

后端数据来源：

- `analysis_records.raw_text`
- `analysis_records.clean_text`
- `analysis_records.sentiment`
- `analysis_records.product_id`

成员 C 处理逻辑：

1. 从历史分析记录中读取评论文本。
2. 对评论进行分词，推荐优先使用 `jieba`；如果依赖缺失，则使用关键词规则兜底。
3. 去掉停用词，例如“这个”“真的”“非常”“一个”“没有”“感觉”等。
4. 按 `sentiment=positive` 和 `sentiment=negative` 分开统计词频。
5. 输出 Top-K 高频词，默认 `top_k=30`。

建议新增文件：

```text
model/text_insights.py
model/resources/stopwords_zh.txt
```

建议核心函数：

```python
def extract_keywords(records, top_k=30):
    """
    输入：历史分析记录列表。
    输出：正向词频和负向词频。
    """
```

接口设计：

```http
GET /api/insights/keywords?product_id=phone&top_k=30
```

响应示例：

```json
{
  "success": true,
  "message": "关键词统计成功",
  "data": {
    "product_id": "phone",
    "top_k": 30,
    "positive_words": [
      {"word": "物流", "count": 18, "weight": 0.92},
      {"word": "质量", "count": 15, "weight": 0.81}
    ],
    "negative_words": [
      {"word": "破损", "count": 9, "weight": 0.76},
      {"word": "太贵", "count": 6, "weight": 0.51}
    ]
  },
  "code": 0
}
```

前端展示要求：

- 展示两个词云区域：正向词云、负向词云。
- 正向词云使用绿色/蓝色系，负向词云使用红色/橙色系。
- 鼠标悬停显示 `word`、`count`、`weight`。
- 没有数据时显示空状态，不允许页面报错。

验收标准：

- 上传至少 20 条混合评论后，词云接口能返回正负向词列表。
- 前端能渲染正向词云和负向词云。
- 词云结果不能包含大量无意义停用词。

### 3.2 价格/物流/质量评价雷达图

目标：从评论中识别用户关注维度，生成商品多维口碑雷达图。

第二阶段先做 3 个核心维度：

| 维度 | 关键词示例 |
| --- | --- |
| 价格 | 价格、便宜、贵、划算、性价比、优惠、值 |
| 物流 | 物流、快递、发货、配送、到货、包装、速度 |
| 质量 | 质量、做工、耐用、正品、破损、坏、材质 |

成员 C 评分逻辑：

1. 遍历某商品所有评论。
2. 判断评论是否命中某个维度关键词。
3. 如果命中，则根据情感结果给该维度加分或减分。
4. 输出每个维度的 `score`、`positive_count`、`negative_count`、`total_count`。

推荐评分公式：

```text
aspect_score = round(100 * positive_count / total_count, 2)
```

如果某个维度没有命中任何评论：

```json
{
  "score": 0,
  "total_count": 0,
  "status": "no_data"
}
```

建议核心函数：

```python
def score_aspects(records):
    """
    输入：历史分析记录列表。
    输出：价格、物流、质量三个维度的评分和命中统计。
    """
```

接口设计：

```http
GET /api/insights/aspects?product_id=phone
```

响应示例：

      }
    ]
  },
  "code": 0

前端展示要求：

- 使用 ECharts 雷达图展示“价格、物流、质量”三个维度。
- 雷达图旁边展示每个维度的命中评论数。
- 如果某个维度 `total_count=0`，显示“暂无评论覆盖该维度”。

验收标准：

- 测试数据中包含“价格、物流、质量”关键词时，接口能返回三个维度评分。
- 前端雷达图可以正常渲染，不出现空白或 NaN。
- 负面评论较多的维度得分应明显低于正面评论较多的维度。

### 3.3 智能避坑助手/一键总结

目标：对某商品的评论做自动总结，输出优点、缺点和购买建议。

该功能为 P2 选做。必须遵守两个原则：

1. 不能强依赖 LLM；没有 API Key 时也要有规则兜底。
2. 摘要不能凭空编造，只能基于已有评论、关键词和维度评分生成。

后端接口：

```http
GET /api/summary/product?product_id=phone
```

响应示例：

```json
{
  "success": true,
  "message": "总结生成成功",
  "data": {
    "product_id": "phone",
    "mode": "rule_fallback",
    "advantages": [
      "多数用户认为物流速度较快",
      "正向评论中频繁出现质量、包装等关键词",
      "价格维度整体评价较好"
    ],
    "disadvantages": [
      "部分用户反馈包装破损",
      "质量维度存在少量负向评论",
      "少数评论提到价格偏高"
    ],
    "buying_advice": "适合重视物流和性价比的用户购买；如果非常关注做工细节，建议结合负向评论进一步判断。",
    "evidence": {
      "positive_words": ["物流", "质量", "划算"],
      "negative_words": ["破损", "太贵"],
      "aspect_scores": {
        "价格": 75.0,
        "物流": 83.33,
        "质量": 60.0
      }
    }
  },
  "code": 0
}
```

实现方案：

- 默认模式：规则摘要。
- 可选模式：LLM 摘要。
- 环境变量建议：

```text
SUMMARY_MODE=rule
LLM_API_KEY=
LLM_BASE_URL=
LLM_MODEL=
```

规则摘要最小实现：

1. 取正向 Top 3 高频词生成优点。
2. 取负向 Top 3 高频词生成缺点。
3. 根据雷达图最低分维度生成避坑建议。
4. 根据雷达图最高分维度生成购买推荐。

验收标准：

- 没有 LLM Key 时接口也能返回总结。
- 返回内容必须包含 `advantages`、`disadvantages`、`buying_advice`、`evidence`。
- 前端能展示总结卡片。

| 词云接口 | 返回正向/负向词频 Top-K | 必须 |
| 词云前端 | 页面能展示正负词云 | 必须 |
| 雷达接口 | 返回价格/物流/质量评分 | 必须 |
| 雷达前端 | 页面能展示雷达图 | 必须 |
| 总结接口 | 无 LLM Key 时规则兜底 | 选做 |
| 总结前端 | 展示优点、缺点、购买建议 | 选做 |
| 性能报告 | 包含平均耗时、P95、缓存命中效果 | 必须 |
| 演示数据 | 30 条以上，可稳定展示效果 | 必须 |

## 9. 风险与规避

| 风险 | 表现 | 规避方式 |
| --- | --- | --- |
| 分词效果差 | 词云全是无意义词 | 加停用词表和领域词典 |
| 雷达图无数据 | 某些维度显示 0 或 NaN | 后端返回 `status=no_data`，前端显示空状态 |
| LLM 不稳定 | 接口超时、没 Key、生成内容乱说 | 摘要设为选做，默认规则兜底 |
| 接口字段变动 | 前端联调失败 | B 先固定响应结构，再给 A 接入 |
| Docker 构建慢 | 拉大模型依赖 | Docker 继续使用轻量 API 依赖，本地模型训练单独执行 |
| 展示数据不足 | 词云/雷达图不好看 | D 准备专用演示 CSV |

## 10. 最终答辩表达建议

第二阶段可以这样讲：

> 第一阶段我们完成了情感分类系统的基本闭环，可以判断评论是正向还是负向。第二阶段我们进一步做了评论洞察能力，不只告诉用户“好评还是差评”，还通过正负向词云展示用户具体在夸什么、吐槽什么；通过价格、物流、质量雷达图展示商品在不同维度上的口碑差异；最后通过规则摘要或 LLM 摘要生成购买建议，让系统从分类工具升级为商品评价分析助手。
