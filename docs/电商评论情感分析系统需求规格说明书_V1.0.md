## 1. 引言

### 1.1 编写目的
本文档旨在全面、客观地定义“电商评论情感分析系统”的系统功能、非功能需求、架构设计及数据规范。本文档将作为后续数据采集、模型训练、后端开发、前端页面构建及容器化部署的基准验收依据。

### 1.2 项目背景
随着电子商务业务规模的不断扩张，用户评价（UGC）产生的数据量呈指数级增长。传统依赖人工抽检的口碑维护机制已无法满足时效性与覆盖率要求。本系统基于深度学习（Deep Learning）与自然语言处理（NLP）技术，通过轻量级预训练语言模型构建一套自动化的文本情感判别引擎，并提供可视化数据看板，赋能商业决策与舆情监控。

### 1.3 关键术语
*   **NLP (Natural Language Processing)**：自然语言处理。
*   **BERT**：基于 Transformer 的双向编码器表征模型。
*   **ETL (Extract, Transform, Load)**：数据提取、转换和加载的流程。
*   **Stratified Sampling**：分层抽样，确保各子集类别比例与全集一致。

## 2. 总体架构设计

本系统采用松耦合的面向服务架构（SOA），具体划分为以下五层数据与业务链路：

1.  **数据接入层**：支持网络爬虫异步抓取数据及本地 CSV 文件的二进制容错解析（兼容 ANSI/GBK 混合编码）。
2.  **ETL 预处理层**：执行定制化正则清洗流水线、数据去噪、中文分词（Jieba）及标签映射。
3.  **核心推理层**：加载经过量化压缩（Quantization）的 `bert-base-chinese` 模型，执行二分类推理及强度打分。
4.  **服务与调度层**：基于 FastAPI 构建 RESTful API 集群，利用 Redis 提供高频结果的内存缓存机制，MySQL 负责持久化落地。
5.  **应用表现层**：采用 Vue3 + ECharts 驱动的前端面板，提供多维度的数据可视化。

## 3. 功能性需求说明 (Functional Requirements)

### 3.1 数据采集与预处理模块 (FR-01)
*   **FR-01-01 定向数据采集**：系统需具备电商短评、追评数据的采集能力，支持网络抓取与本地全量数据集导入。
*   **FR-01-02 自动化数据清洗**：系统需按照严格的执行顺序清洗原始文本（包含剥离 URL 链接、清除全部空白字符、滤除非中文字符）。
*   **FR-01-03 异常值与噪声过滤**：自动识别并剔除空文本、长度阈值异常样本（长度 < 2）、重复样本及中性情感样本。
*   **FR-01-04 智能分词与抽样**：集成 Jieba 精确模式分词引擎；具备按照 8:1:1 比例对数据集进行分层抽样（Stratified Split）的能力，以输出训练集、验证集与测试集。

### 3.2 NLP 核心推理模块 (FR-02)
*   **FR-02-01 情感极性判别**：能够精准识别评论的正向（Label=1）与负向（Label=0）情感极性。
*   **FR-02-02 情感强度量化**：基于分类置信度或回归计算，输出 0-10 范围内的连续情感强度分值。
*   **FR-02-03 CPU 推理加速**：模型需经过结构化优化或量化处理，以降低边缘侧/无 GPU 环境下的推理延迟。

### 3.3 API 服务与数据流转模块 (FR-03)
*   **FR-03-01 在线推理接口**：提供单条文本及 Excel/CSV 批量预测的标准化 HTTP 接口。
*   **FR-03-02 高并发缓存处理**：针对重复的查询请求，引入 Redis 缓存层，规避底层模型重复计算。
*   **FR-03-03 历史检索与导出**：支持对持久化存储的分析记录进行条件分页查询，并支持结果报表的 Excel 导出。

### 3.4 终端交互与可视化模块 (FR-04)
*   **FR-04-01 交互式分析工作台**：提供所见即所得的单条/批量文件测试入口。
*   **FR-04-02 BI 数据看板**：动态渲染情感正负面分布图（饼图）与时序口碑波动趋势图（柱状图/折线图）。

## 4. 数据需求与存储规范 (Data Specifications)

### 4.1 输入数据约束
*   **文件格式要求**：系统需兼容本地 `shopping_comments.csv` 格式输入。
*   **编码容错标准**：系统输入模块需配置二进制读取与 `GBK` 容错解码策略（`errors="replace"`），以规避非标准 Unicode 字符引发的程序崩溃。

### 4.2 标准化数据字典 (Data Dictionary)
经预处理层清洗输出的规范化数据集文件需统一采用 `utf-8-sig` 编码格式，以保障跨平台（含 Windows Excel）的兼容性。落盘字段规范如下：

| 字段名称 | 物理类型 | 逻辑描述 | 约束说明 |
| :--- | :--- | :--- | :--- |
| `product_id` | String | 商品大类 ID | 映射自原始数据 `cat` 字段 |
| `content` | String | 原始评论文本 | 保留原始文本作为溯源比对基准 |
| `comment_time` | Date | 评论时间戳 | 标准化格式：YYYY-MM-DD |
| `star` | Integer | 商品评级星级 | 1 对应 1星(差评)；5 对应 5星(好评) |
| `label` | Integer | 情感分类标签 | 离散值：0 (负向)，1 (正向) |
| `clean_content`| String | 清洗后文本 | 纯中文字符，无空格/URL/特殊符号 |
| `cut_words` | String | 预分词序列 | 以半角空格分隔，服务于模型输入 |

### 4.3 预处理流水线执行流 (Pipeline Sequence)
文本预处理具备严格的原子性执行时序，不可逆调换：
1. **RegEx 过滤**：移除 `HTTP/HTTPS` 等完整 URL 协议特征。
2. **Space 过滤**：清除制表符、全/半角空格及换行符（`\s`）。
3. **Char 过滤**：基于 Unicode 编码区间 `\u4e00-\u9fff`，剔除非中文字符。
4. **Trim 处理**：对清洗末端字符串执行首尾空白裁切。

## 5. 非功能性需求 (Non-Functional Requirements)

### 5.1 性能与响应需求
*   **推理延迟（Latency）**：单条文本请求的端到端接口响应时间应 ≤ 500ms（依托轻量化模型部署实现）。
*   **吞吐量与并发（Throughput）**：结合 FastAPI 的异步（Asynchronous）特性及 Redis 内存缓存架构，系统需稳定支撑高频并发调用，降低计算图的直接访问压力。

### 5.2 可靠性与容错需求
*   **编码异常容错**：数据摄入模块必须具备容错机制，对乱码或无法识别的生僻字符执行替换或丢弃策略，严禁引发主进程崩溃（Crash）。
*   **服务降级**：当深度学习计算节点资源耗尽时，API 网关应通过捕获异常并返回标准化的 HTTP 错误码（如 503），保证前端系统的高可用与友好交互。

### 5.3 部署与运维需求
*   **环境隔离编排**：全栈项目组件（含 Vue3 前端、Python FastAPI 后端、MySQL 数据库、Redis 服务）需配置标准化的 `Dockerfile`。
*   **一键式交付**：依托 `docker-compose.yml` 脚本，实现无视宿主机底层环境的“一键拉起”与“统一调度”部署机制。

### 5.4 实验可复现性（Reproducibility）
*   在执行数据集切分（Train/Validation/Test）以及模型权重初始化阶段，系统代码必须设定全局统一的随机数种子（如 `random_state=42`），以保障模型评估指标与实验结论的绝对客观性与可复现性。

## 6. 项目交付物与验收标准 (Deliverables & Acceptance Criteria)

### 6.1 核心交付产物
1.  **数据资产**：标准化输出的 4 份 UTF-8-SIG 编码的 CSV 数据集（`cleaned_full_data.csv`, `train.csv`, `val.csv`, `test.csv`）。
2.  **代码资产**：涵盖数据清洗脚本、BERT 微调脚本、FastAPI 后端源码、Vue3 前端工程源码。
3.  **模型资产**：训练完毕并经量化压缩的情感二分类模型权重文件。
4.  **工程资产**：Docker 配置编排文件及数据库 DDL 初始化脚本。

### 6.2 核心验收基准
1.  **数据指标要求**：随机抽检输出数据集的 `clean_content` 字段，需达到 100% 纯中文且无残留噪声；各切分子集的正负样本比例必须与全量清洗数据集严格保持一致。
2.  **模型精度要求**：在独立划分的测试集（Test Set）上，情感分类模型的精确率、召回率与 F1-Score 需达到可用于业务决策的标准基线。
3.  **系统连通性要求**：在前端交互面板上传包含混合评论的测试文件，需证明前后端全链路运转畅通，正确渲染 ECharts 统计图表，且分析记录已成功写入持久化数据库。
